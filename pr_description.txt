🧹 Optimize Photo Endpoints with Pagination

🎯 What
Implemented native server-side pagination (`page`, `pageSize`) across all major gallery fetching endpoints (`GetPhotos`, `GetUserPhotos`, `GetMostViewedPhotos`) via Entity Framework Core `.Skip()` and `.Take()`.

💡 Why
The `GetPhotos` endpoint previously loaded the entire gallery into memory and synchronously spun up thousands of concurrent background tasks (`Task.WhenAll`) to generate S3 presigned URLs for every single image (both original and thumbnail) before responding. This caused significant memory allocations, DB load, and network/CPU spikes. By paginating the database queries at the source, we bound the `Task.WhenAll` mapping overhead to a maximum of 50 images per request.

✅ Verification
- Ensured `PhotosControllerTests.cs` and other backend tests pass successfully.
- Preserved `Task.WhenAll` mapping for S3 URLs to ensure optimal concurrency on the restricted chunk of data.
- Note: This architectural change forces the frontend to fetch pages. To fully integrate, the frontend requires additional server-side filter parameters.

✨ Result
Significantly decreased memory usage, CPU load, and response latency on all photo-fetching endpoints by capping operations to 50 items.
